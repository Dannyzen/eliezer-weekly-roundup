## Executive Summary

1. **Agent discovery needs behavioral evidence.** AgentSearchBench shows that semantic similarity between task text and agent descriptions is not enough; agent routing needs execution-aware probing and outcome-based reranking.
2. **Agent memory needs type and version semantics.** Memanto argues that typed categories, temporal versioning, conflict resolution, and low-latency retrieval can beat graph-heavy memory pipelines in practical long-horizon settings.
3. **World modeling gives agents a useful evaluation map, not a turnkey recipe.** The Agentic World Modeling survey separates predictors, simulators, and evolvers across physical, digital, social, and scientific law regimes.
4. **Sovereignty is becoming local context infrastructure.** GitNexus and Beads show a practical local-first substrate for coding agents: code graphs, task graphs, MCP surfaces, and auditable state before cloud escalation.

### 2. Memanto makes typed, versioned memory a practical alternative to graph-heavy agent memory
- Category: [AgenticAI](../AgenticAI/2026-04-27/reasoning.md#memanto-makes-typed-versioned-memory-a-practical-alternative-to-graph-heavy-agent-memory)
- Durable topic: [Memory Systems](../AgenticAI/memory-systems/memory-systems.md)
- Core source: https://arxiv.org/abs/2604.22085
- Why it matters: long-horizon memory fails when writes are untyped, contradictions are unresolved, and retrieval is too expensive to use every turn.
- Implementable now: typed memory categories, temporal versioning, explicit supersession/conflict metadata, LongMemEval/LoCoMo-style tests.
- Implementability score: 0.66

### 3. Agentic world modeling is a useful map, but not yet a turnkey implementation pattern
- Category: [AgenticAI](../AgenticAI/2026-04-27/reasoning.md#agentic-world-modeling-is-a-useful-map-but-not-yet-a-turnkey-implementation-pattern)
- Core source: https://arxiv.org/abs/2604.22748
- Supporting sources: https://github.com/matrix-agent/awesome-agentic-world-modeling, https://github.com/Snowflake-Labs/agent-world-model
- Why it matters: agent builders need to state which environment dynamics their systems can predict, simulate, or revise.
- Implementable now: use the L1/L2/L3 taxonomy as an eval rubric, start with digital state-transition tests, and use synthetic MCP environments for safe rollout evaluation.
- Implementability score: 0.46

### 4. Local-first code and task context is becoming the sovereign coding-agent substrate

- Why it matters: repository context and task state are sensitive operational data. They should live in operator-controlled substrates before being routed to hosted models.

## Signals

- **Highest implementability:** local-first code and task context at 0.86. GitNexus and Beads are installable and operational, though GitNexus licensing and RC status need review before production use.
- **Lowest implementability:** agentic world modeling at 0.46. The taxonomy is immediately useful as a design/eval rubric, but robust simulators and self-revising evolvers are still heavy research/infrastructure work.
- **Most actionable repo move:** add behavioral probes to agent/tool registries and log outcomes as routing data.
- **Most important architecture move:** treat memory writes as typed, versioned state transitions, not append-only vector facts.

## Actionable Insights

1. Build a small probe suite for every reusable agent/tool before routing privileged work to it.
2. Add typed memory categories, version lineage, and conflict status before adding more vector collections.
3. Use world-model vocabulary to scope agent claims: predictor, simulator, or evolver; physical, digital, social, or scientific.
4. Try a local code-context substrate and a graph task tracker on one coding-agent workflow.
5. Route minimal retrieved context to hosted models; keep repo graphs and task history local unless escalation is justified.
