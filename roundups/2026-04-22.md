# Daily Scan: 2026-04-22

Today’s strongest signal is that the serious agent stack is getting more operationally legible. Delegation is moving from fixed role labels to context-conditioned evidence, computer-use reliability is being redefined around repeatability, local-first claims are getting tested by subtask and backend instead of ideology, and high-autonomy security tools are starting to ship with containment baked into the runtime.

## Top findings

### 1. Delegation should be calibrated to context, not static agent roles
- Category: [AgenticAI](../AgenticAI/2026-04-22/reasoning.md#delegation-should-be-calibrated-to-context-not-static-agent-roles)
- Durable topic: [Agent Harness Architecture](../AgenticAI/agent-harness-architecture/agent-harness-architecture.md)
- Core source: https://arxiv.org/abs/2604.17950
- Why it matters: multi-agent systems route badly when they treat “good coder” or “good planner” as a global truth instead of a context-shaped probability estimate
- Implementability score: 0.79

### 2. Repeatability is becoming the real reliability metric for computer-use agents
- Category: [AgenticAI](../AgenticAI/2026-04-22/reasoning.md#repeatability-is-becoming-the-real-reliability-metric-for-computer-use-agents)
- Durable topic: [Trajectory-Aware Evaluation](../AgenticAI/trajectory-aware-evaluation/trajectory-aware-evaluation.md)
- Core source: https://arxiv.org/abs/2604.17849
- Why it matters: a browser or desktop agent that succeeds once and fails on replay is still an unreliable system, not a solved capability
- Implementability score: 0.86

### 3. Local-first claims need task-shaped benchmarking, not slogans
- Category: [Strategy](../Strategy/2026-04-22/sovereignty.md#local-first-claims-need-task-shaped-benchmarking-not-slogans)
- Durable topic: [Local-First Agents](../Strategy/local-first-agents/local-first-agents.md)
- Core source: https://arxiv.org/abs/2604.18566
- Why it matters: structured extraction is much closer to local-ready than long-context coaching or error fixing, so local-first has to be routed by workload shape
- Implementability score: 0.90

### 4. High-autonomy security agents are forcing containment into the product surface
- Category: [Strategy](../Strategy/2026-04-22/sovereignty.md#high-autonomy-security-agents-are-forcing-containment-into-the-product-surface)
- Durable topic: [Agent Sandboxing](../Strategy/agent-sandboxing/agent-sandboxing.md)
- Core source: https://github.com/KeygraphHQ/shannon/releases/tag/v1.1.0
- Supporting source: https://github.com/KeygraphHQ/shannon
- Why it matters: agents that can inspect code and execute real exploits now need read-only mounts, writable overlays, and self-hosted boundaries as default runtime assumptions
- Implementability score: 0.92

## What is implementable now

The clearest moves you can make now are:
- route multi-agent work using per-context performance histories instead of one static specialist score
- evaluate browser and desktop agents under repeated execution before trusting a pass rate
- split local-first workloads by task shape so structured extraction stays local while long-context correction escalates selectively
- treat sandbox topology and writable boundaries as part of the product whenever an agent can execute high-risk actions

## Opinionated read

The field is getting less theatrical and more accountable. That is good. The next edge will come from teams that can say which agent should act on this task, how often that behavior repeats, which subtasks truly deserve cloud escalation, and exactly what boundary keeps a high-autonomy agent from becoming an uncontrolled shell.
